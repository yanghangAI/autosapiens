## 2. Flexible `data_root` via `BEDLAM2_DATA_ROOT` env var

**Problem:** `data_root` was hardcoded to a server-specific path in the config and shell
scripts, breaking portability across machines.

**Solution:** All scripts now read `BEDLAM2_DATA_ROOT` from the environment, with the
previous path as fallback.

**Usage:**
```bash
export BEDLAM2_DATA_ROOT=/your/path/to/bedlam2
bash pose/scripts/finetune/bedlam2/sapiens_0.3b/node.sh
```

| File | Change |
|------|--------|
| `pose/configs/sapiens_pose/bedlam2/sapiens_0.3b-50e_bedlam2-640x384.py` | `data_root = os.path.expanduser(os.environ.get('BEDLAM2_DATA_ROOT', '~/repos_local/MMC/BEDLAM2Datatest'))` |
| `pose/scripts/finetune/bedlam2/sapiens_0.3b/node.sh` | Added `DATA_ROOT="${BEDLAM2_DATA_ROOT:-}"` variable; passes `data_root=$DATA_ROOT` via `--cfg-options` when set |
| `pose/scripts/demo/local/bedlam2.sh` | `DATA_ROOT="${BEDLAM2_DATA_ROOT:-/media/s/SF_backup/bedlam2}"` |
| `docs/BEDLAM2_TRAINING.md` | Replaced hardcoded paths with `export BEDLAM2_DATA_ROOT=...` instruction |

## 3. Bug fixes to get `pose/tools/train.py` running

Multiple issues discovered and fixed when running `pose/tools/train.py` for the first time
in the `sapiens` conda environment.

### 3a. `yapf` version incompatibility in MMEngine
**File:** `engine/mmengine/config/config.py:1481`
**Fix:** Removed `verify=True` from `FormatCode()` call — this argument was removed in
yapf ≥ 0.40 but MMEngine 1.0.0 still passes it.

### 3b. Missing Python dependencies in `sapiens` env
Installed the following packages that were missing:
```bash
pip install xtcocotools scipy pandas munkres
```
- `xtcocotools` — required by COCO-style dataset base classes in mmpose
- `scipy` — required by `MpiiDataset`
- `pandas` — required by `Goliath3dDataset`
- `munkres` — required by mmpose evaluation utilities

**Note:** numpy must be pinned to `1.26.4` — xtcocotools binary is compiled against
numpy 1.x ABI. Upgrading to numpy 2.x breaks xtcocotools with a binary incompatibility.

### 3c. mmpose not installed as a package
**Problem:** mmpose was only accessible because `pose/` was implicitly on sys.path when
running from that directory. `custom_imports` in MMEngine configs runs in a temp dir
where `pose/` is not on sys.path, so `import mmpose` failed.
**Fix:**
```bash
pip install -e pose/ --no-deps
```

### 3d. `custom_imports` added to BEDLAM2 config
**File:** `pose/configs/sapiens_pose/bedlam2/sapiens_0.3b-50e_bedlam2-640x384.py`
Added `custom_imports` block to force-import all BEDLAM2-specific modules so they
register with MMEngine before the Runner builds models/datasets. Without this, classes
like `RGBDPose3dEstimator` are never registered in mmpose's registry and the runner fails
with `KeyError`.

### 3e. `Visualizer` override in BEDLAM2 config
**File:** `pose/configs/sapiens_pose/bedlam2/sapiens_0.3b-50e_bedlam2-640x384.py`
Overrode the inherited `PoseLocalVisualizer` (which requires the full mmpose import chain)
with the base `Visualizer` from mmengine, since 2D pose visualization is not needed for
the 3D BEDLAM2 task.

### 3f. `RGBDPose3dEstimator.loss()` returned a tuple instead of dict
**File:** `pose/mmpose/models/pose_estimators/rgbd_pose3d.py`
**Fix:** Changed `return losses, preds` → `return losses`. MMEngine's `BaseModel.parse_losses()`
expects a plain `dict` from `loss()`, not a `(dict, dict)` tuple.

### 3g. Tilde `~` not expanded in `data_root`
**File:** `pose/configs/sapiens_pose/bedlam2/sapiens_0.3b-50e_bedlam2-640x384.py`
**Fix:** Wrapped `data_root` assignment with `os.path.expanduser()` so `~/...` paths
are correctly resolved at config parse time.

## 4. Bug fix: `frame_stride=5` on already-6fps frames

### `pose/configs/sapiens_pose/bedlam2/sapiens_0.3b-50e_bedlam2-640x384.py`
Fixed `frame_stride=5` → `frame_stride=1` in all three dataloaders (train, val, test).

**Why:** `extract_frames.py` already downsamples from 30fps to 6fps at extraction time — JPEGs on disk are named `00000.jpg`, `00001.jpg`, ... where each represents every 5th video frame. The NPZ `n_frames` field stores the 6fps frame count. Applying `frame_stride=5` on top was sub-sampling the already-6fps frames, discarding 80% of training data.

**Effect:** Training samples corrected from 21,753 → 104,114 (train), 2,780 → 13,293 (val). Batches/epoch now matches `claude_code/train.py` (6,508 batches × batch_size=16).

## 7. Pose3dVisualizationHook verification and bug fixes

Ran a 5-epoch test training on a tiny dataset (3 train seqs, 2 val seqs) to verify the
BEDLAM2 video visualization. Found and fixed 3 bugs:

### 7a. `after_train_iter` crash: `KeyError: 'vis_preds'`
**File:** `pose/mmpose/engine/hooks/pose3d_visualization_hook.py`
**Problem:** When `enable=True` (set for BEDLAM2 video mode), the original heatmap-mode
`after_train_iter` ran and tried to access `outputs['vis_preds']['pose2d']`, which doesn't
exist in the BEDLAM2 model output.
**Fix:** Added `self.bedlam2_video` guard to skip `after_train_iter` in BEDLAM2 mode.

### 7b. TensorBoard writer not found
**File:** `pose/mmpose/engine/hooks/pose3d_visualization_hook.py`
**Problem:** `runner.visualizer.get_backend('TensorboardVisBackend')` returned `None` due to
the `@master_only` decorator and attribute name mismatch.
**Fix:** Access `runner.visualizer._vis_backends` dict directly, force-init env, and return
`_tensorboard` attribute.

### 7c. `dataset.data_list` empty after serialization
**File:** `pose/mmpose/engine/hooks/pose3d_visualization_hook.py`
**Problem:** MMEngine's `BaseDataset._serialize_data()` calls `self.data_list.clear()` for
shared-memory efficiency. All visualization methods that iterated `dataset.data_list` got
empty lists.
**Fix:** Replaced all `dataset.data_list[i]` accesses with `dataset.get_data_info(i)`,
which deserializes from the `data_bytes` array. Added `_bedlam2_get_data_list()` helper
for methods that need the full list.

### Verification result
5-epoch test completed successfully. TensorBoard logs contain 10 image tags
(val/train × 3 scenes × gt_pelvis/pred_pelvis), updated each epoch.
Training: loss 4.65 → 1.44, MPJPE 1099 → 359mm over 5 epochs.
